# 技術仕様設計ワークフロー

このスキルは要件定義書から技術仕様書を生成するプロセスを定義する。

## セットアップ

フィーチャー名はスキルの引数として渡される。
フィーチャー名が指定されていない場合は、作業前に呼び出し元へ確認する。

> 「どのフィーチャーの仕様書を設計しますか？」

フィーチャー名が確定したら、要件定義書の存在を確認する。

- `.artifacts/requirements/req_[feature_name].md` が **存在する** 場合: Phase 1 へ進む。
- **存在しない** 場合: 呼び出し元へ通知し、対応を確認する。

> 「要件定義書 `req_[feature_name].md` が見つかりません。
> 先に `/req [feature_name]` を実行して作成するか、
> 要件をインラインで提供してください。」

有効な要件定義書が用意されるか、呼び出し元がインラインで要件を提供するまで作業を進めない。

## ワークフロー

### Phase 1: 現状分析

- `.artifacts/requirements/req_[feature_name].md` を読み込む
- 既存のコードベースがある場合は `Grep` / `Glob` でアーキテクチャ・パターン・規約を把握する
- 技術的負債やスケーラビリティの制約があれば記録する
- `.claude/.works/research/*.md` が存在する場合は読み込み、調査結果を設計へ反映する

### Phase 2: 設計

仕様書で以下の領域を扱う。

#### 機能要件

- **技術スタック**: 言語・フレームワーク・外部サービスの選定と根拠
- **アーキテクチャ**: 全体構造・データフロー・提案するディレクトリ構成
- **データモデル / スキーマ**: エンティティ / テーブル定義・型定義・バリデーションルール
- **インターフェース仕様**: API エンドポイント・関数シグネチャ・状態管理方針
- **UI/UX コンポーネント**: 主要コンポーネント一覧・画面遷移・インタラクション詳細（該当する場合）
- **実装ステップ**: タスクレベルの実装計画
- **テスト戦略**: 単体テスト対象・ハッピーパス / エラーパスの定義・テストコマンド例

#### 非機能要件

- **セキュリティ**: 多層防御・最小権限の原則・境界での入力バリデーション
- **パフォーマンス**: 効率的なアルゴリズム・クエリ最適化・キャッシュ戦略
- **スケーラビリティ**: 水平スケーリングの可否・ステートレス設計の範囲
- **保守性**: 一貫したパターン・テスタビリティ・明確なコード構成

#### トレードオフ分析

重要な設計決定については以下を記録する。

- **メリット**: 利点と優位性
- **デメリット**: 欠点と制限
- **代替案**: 検討した他の選択肢
- **決定**: 最終選択とその根拠

#### 出力前の確認（必要な場合）

要件定義書を分析した後も曖昧な点や不足している要件がある場合は、
出力前に呼び出し元へ確認する。

> **ツール使用**: 確認が必要な場合は `AskUserQuestion` ツールを使用する。
> 質問をプレーンテキストとしてインラインに書かず、常に `AskUserQuestion` を使用することで
> 呼び出し元が明確に回答でき、会話が整理された状態を保つ。

### Phase 3: 出力

`Read` で `./template.md` のテンプレートを読み込む。
次に `Write`（新規ファイル）または `Edit`（既存ファイル）で
`.artifacts/specifications/spec_[feature_name].md` を作成または更新する。

#### 図

以下のいずれかに該当する場合は Mermaid 図を作成する。

- 全体アーキテクチャ構造（レイヤー図・コンポーネント図）
- データフロー（シーケンス図・フロー図）
- データモデル（ER 図・クラス図）
- 画面遷移図

### Phase 4: 報告

設計内容をサマリーとして報告し、`.artifacts/specifications/spec_[feature_name].md` に記録したことを呼び出し元へ伝える。
実装計画フェーズへ進むために `/plan` コマンドの実行を促す。

## ベストプラクティス

- 要件で「未決定」のままになっている項目は、設計上の前提として明示的に記録する
- 過剰設計を避ける — 現在の要件に必要なものだけ設計し、投機的な将来の拡張性は考慮しない
- アンチパターンを検出した場合は明示的にフラグを立てる（以下のアンチパターン警告を参照）

## アーキテクチャ原則

- **モジュール性と関心の分離**: 単一責任・高凝集 / 低結合・明確なインターフェース
- **スケーラビリティ**: 水平スケーリング能力・可能な限りステートレス設計
- **保守性**: 明確なコード構成・一貫したパターン・テストのしやすさ
- **セキュリティ**: セキュア・バイ・デフォルト・最小権限の原則・監査証跡
- **パフォーマンス**: ネットワークリクエストの最小化・適切なキャッシュ・遅延ロード

## アンチパターン警告

設計中に以下を検出した場合はフラグを立てて対処する。

- **Big Ball of Mud**: 明確な構造がない
- **Golden Hammer**: 同じ解決策をあらゆる問題に適用する
- **Premature Optimization**: 計測なしに最適化する
- **Tight Coupling**: コンポーネント間の過度な依存
- **God Object**: 1 つのクラス / モジュールがすべてを担う
- **Analysis Paralysis**: 過剰な計画で実装が進まない
