# フロントエンド リッチプレゼンテーション

ブラウザ上で完結する、アニメーション豊かなゼロ依存 HTML プレゼンテーションを作成する。

ビジュアル探索アプローチにインスパイアされた作品（クレジット: @zarazhangrui）。

## アクティブ化タイミング

- トークデッキ、ピッチデッキ、ワークショップ、社内資料の HTML 化
- `.ppt` / `.pptx` スライドを HTML プレゼンテーションへ変換
- 既存 HTML プレゼンテーションのレイアウト・モーション・タイポグラフィの改善
- デザイン好みが不明なユーザーとプレゼンテーションスタイルを探索する場合

## 絶対守るべきルール

1. **ゼロ依存**: デフォルトはインライン CSS・JS を含む単一 HTML ファイル。
1. **ビューポートフィットは必須**: 各スライドは 1 ビューポートに完全に収まり、内部スクロールは不可。
1. **説明でなく見せる**: 抽象的なスタイルアンケートでなくビジュアルプレビューを使う。
1. **個性的なデザイン**: 一般的なパープルグラデーションや Inter 字体のテンプレート風は避ける。
1. **本番品質**: コードにコメントを付け、アクセシブル・レスポンシブ・パフォーマンス重視で作成する。

生成前に `frontend-styles.md` を読み、ビューポートセーフな CSS ベース、密度上限、プリセットカタログ、CSS 注意点を確認する。

## ワークフロー

### 1. モード判定

いずれかのパスを選択する:

- **新規プレゼンテーション**: ユーザーがトピック、メモ、あるいは完成原稿を持っている場合
- **PPT 変換**: ユーザーが `.ppt` / `.pptx` を持っている場合
- **改善**: ユーザーがすでに HTML スライドを持ち、改善を希望する場合

### 2. コンテンツの確認

最低限必要なことだけ確認する:

- 目的: ピッチ、教育、カンファレンス登壇、社内報告
- 分量: 短（5-10 枚）、中（10-20 枚）、長（20 枚以上）
- コンテンツ状態: 完成原稿、大まかなメモ、トピックのみ

ユーザーがコンテンツを持っている場合は、スタイリング前に貼り付けてもらう。

### 3. スタイルの確認

ビジュアル探索をデフォルトとする。

ユーザーがプリセットを指定済みの場合、プレビューをスキップしてそのまま使う。

それ以外の場合:

1. デッキが作るべき印象を確認する（例: 感心、活力、集中、インスピレーション）。
1. `.ecc-design/slide-previews/` に **3 枚の単一スライドプレビューファイル** を生成する。
1. 各プレビューは完全自包含で、タイポグラフィー・カラー・モーションを明確に示し、スライドコンテンツはおおよそ 100 行以内にする。
1. ユーザーにどのプレビューを残すか、どの要素を組み合わせるかを確認する。

ムードからスタイルへのマッピングは `frontend-styles.md` のプリセットガイドを参照する。

### 4. プレゼンテーションの構築

出力形式:

- `presentation.html`
- `[プレゼンテーション名].html`

デッキに抽出・ユーザー提供画像が含まれる場合のみ `assets/` フォルダを使う。

必須構造:

- セマンティックなスライドセクション
- `frontend-styles.md` のビューポートセーフ CSS ベース
- テーマ値用の CSS カスタムプロパティ
- キーボード・ホイール・タッチナビゲーション対応のプレゼンテーションコントローラークラス
- リビールアニメーション用 Intersection Observer
- reduced-motion サポート

### 5. ビューポートフィットの強制

これはハードゲートとして扱う。

ルール:

- すべての `.slide` に `height: 100vh; height: 100dvh; overflow: hidden;` を適用する
- すべての文字・余白は `clamp()` でスケールさせる
- コンテンツが収まらない場合は複数スライドに分割する
- 読めないサイズまで文字を小さくしてオーバーフローを解決しない
- スライド内のスクロールバーを許さない

密度上限と必須 CSS ブロックは `frontend-styles.md` を参照する。

### 6. 検証

以下のサイズで完成デッキを確認する:

- 1920x1080
- 1280x720
- 768x1024
- 375x667
- 667x375

ブラウザ自動化が利用可能な場合は、スライドのオーバーフローとキーボードナビゲーションを自動検証する。

### 7. 納品

ハンドオフ時:

- ユーザーが保持を希望しない限り、一時プレビューファイルを削除する
- 必要に応じてプラットフォーム標準のオープナーでデッキを開く
- ファイルパス、使用プリセット、スライド枚数、テーマカスタマイズポイントを要約する

OS 別の正しいオープナーを使う:

- macOS: `open file.html`
- Linux: `xdg-open file.html`
- Windows: `start "" file.html`

## PPT / PPTX 変換

PowerPoint 変換の手順:

1. `python-pptx` を使った `python3` でテキスト・画像・ノートを抽出する。
1. `python-pptx` が利用不可な場合は、インストールするか手動エクスポートベースのワークフローにフォールバックするかユーザーに確認する。
1. スライド順序、スピーカーノート、抽出アセットを保持する。
1. 抽出後は新規プレゼンテーションと同じスタイル選択ワークフローを実行する。

Cross-Platform 対応を維持する。Python で実現できる場合は macOS 専用ツールに依存しない。

## 実装要件

### HTML / CSS

- ユーザーが明示的にマルチファイル構成を求めない限り、インライン CSS と JS を使う。
- フォントは Google Fonts または Fontshare から読み込む。
- 雰囲気のある背景、強い文字階層、明確なビジュアル方向性を優先する。
- イラストでなく抽象形、グラデーション、グリッド、ノイズ、ジオメトリを使う。

### JavaScript

以下を必ず含める:

- キーボードナビゲーション
- タッチ / スワイプナビゲーション
- マウスホイールナビゲーション
- プログレスインジケーターまたはスライドインデックス
- エンター時のリビールアニメーショントリガー

### アクセシビリティ

- セマンティック構造を使う（`main`、`section`、`nav`）
- コントラストを読めるレベルに維持する
- キーボードのみのナビゲーションをサポートする
- `prefers-reduced-motion` を尊重する

## コンテンツ密度上限

ユーザーが明示的に密度を追加しても読みやすさが満たされる場合を除き、以下を上限とする:

| スライド種別         | 上限                                                 |
| -------------------- | ---------------------------------------------------- |
| タイトル             | 見出し 1 個 + サブタイトル 1 個 + タグライン（任意） |
| コンテンツ           | 見出し 1 個 + 箇条書き 4-6 個 or 短文 2 段落         |
| フィーチャーグリッド | カード 6 枚まで                                      |
| コード               | 8-10 行まで                                          |
| 引用                 | 引用 1 個 + 帰属                                     |
| 画像                 | ビューポートに収まる画像 1 枚                        |

## アンチパターン

- ビジュアルアイデンティティのない一般的なスタートアップグラデーション
- 意図的に編集的な印象を明確に希望しない限り、システムフォントのデッキ
- 箇条書きの壁、小さすぎる文字、スクロールが必要なコードブロック
- 短い画面で崩れる固定高さのコンテンツボックス
- `-clamp(...)` などの無効な CSS 関数否定

## 納品チェックリスト

- プレゼンテーションがブラウザでローカルファイルから動作する
- すべてのスライドがスクロールなしでビューポートに収まる
- スタイルが個性的で意図的である
- アニメーションが意味を持ち、雑音になっていない
- reduced-motion が尊重されている
- ファイルパスとカスタマイズポイントが納品時に説明されている
