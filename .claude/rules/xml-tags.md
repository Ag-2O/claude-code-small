# XML タグの取り扱い

`.claude/` 配下のエージェント定義・スキル・スラッシュコマンド、および `CLAUDE.md` のように
**指示・コンテキスト・例・出力形式が混在するプロンプトファイル**では、情報境界を明確化するために
XML タグでセクションを構造化する。Anthropic 公式のプロンプトベストプラクティス
（[Structure prompts with XML tags](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices#structure-prompts-with-xml-tags)）
に沿った実践とする。

通常の Markdown ドキュメント（仕様書・README・成果物）には適用しない。それらは
`.claude/rules/markdown.md` の汎用ルールに従う。

## 適用範囲と禁止領域

- **本文では積極的に使う**: `.claude/agents/`・`.claude/skills/`・`.claude/commands/`・`CLAUDE.md` の本文。
- **YAML フロントマター（`---` 内）では絶対に使わない**: `name` や `description` に山括弧を入れると、
  Claude Code のシステムプロンプトの構造を破壊し、自動ルーティングが失敗する致命的エラーになる。
- 通常の Markdown ドキュメント（仕様書・README・成果物）では使わない。

## 正準タグの一覧

このプロジェクトで使える XML タグは次の正準セットに限定する。**この一覧にないタグを新設しない。**
細かい構造（手順の下位区分・補足ブロック）は、独自タグを生やさず `##` / `###` の Markdown 見出しで表現する。

| タグ           | 用途                                                           | 主な適用先    |
| -------------- | -------------------------------------------------------------- | ------------- |
| `<role>`       | エージェントの役割・ペルソナの定義                             | agents のみ   |
| `<inputs>`     | 引数や入力パラメータの構造化                                   | agents/skills |
| `<workflow>`   | 実行手順（下位区分は `##` / `###` 見出しで表す）               | agents/skills |
| `<thinking>`   | 推論のステップ展開（出力に含めない内部推論として誘導する場合） | agents/skills |
| `<examples>`   | Few-shot 例の集合                                              | agents/skills |
| `<example>`    | 個別の例（`<examples>` 内にネストする）                        | agents/skills |
| `<report>`     | 呼び出し元への報告フォーマット                                 | agents 中心   |
| `<principles>` | 守るべき原則・制約・スコープ（肯定形でまとめる）               | agents/skills |

### 各タグの使い分け

- **`<role>` は agents 専用とする**: 「役割・ペルソナ」はサブエージェント定義の責務である。
  スキル（`SKILL.md`）には `<role>` を置かない。スキルの目的は frontmatter の `description` と
  H1 直下のプレーンな前書きで示す。
- **スコープ・制約は `<principles>` に集約する**: 担当範囲（何をするか）や制約条件は独立タグにせず、
  `<principles>` の中へ肯定形でまとめる。
- **手順の下位構造は見出しで表す**: フェーズ・ステップ・補足（エッジケース・チェックポイント等）は
  `<workflow>` 内の `##` / `###` 見出しで表現し、`<setup>` や `<tdd-steps>` のような独自タグを作らない。

## 命名規則

- **アンダースコアを含めない**: `<output_format>` のようなタグ名は mdformat が強調記法と衝突判定して
  エスケープしてしまう。代わりに `<report>` のような単一語、または `<output-format>` のような
  kebab-case を使う。
- 小文字英字とハイフンのみで構成する。
- 正準タグの名前をそのまま使う。同じ意味のタグは同じ名前で揃える。
- 自然な階層がある場合はネストする（例: `<examples>` の中に複数の `<example>` を置く）。

## 書式

- **タグの開始行と終了行の前後には必ず空行を入れる**。空行がないと mdformat が前後のリストや段落の
  継続行と誤認し、インデントが崩れたりエスケープされたりする。
- タグ内でリストや見出しを使う場合、リストの最終項目の後にも空行を入れてから閉じタグを書く。
- タグ内のコンテンツは通常の Markdown（リスト・コードブロック・強調）で記述してよい。

## 書き方の指針

- **肯定形で書く**: 「〜しない」より「〜する」の表現を優先する（公式推奨）。禁止事項を独立タグで囲うよりも、
  `<principles>` の中で肯定形に言い換える。
- **出力スタイルは目的に合わせる**: 呼び出し元がマークダウン箇条書きをパースする想定なら、
  `<report>` 内はマークダウン箇条書きで書く。プロンプトの style は出力 style に影響するため、
  XML タグを多用しすぎると Claude の応答も XML 寄りになりやすい。
- **横断ルールはタグ内で完結させる**: Claude はリテラルに解釈するため、特定タグ内の指示が
  別のタグやフェーズに自動的に伝播しない。複数フェーズに適用すべきルールは各タグ内で繰り返すか、
  最上位の `<principles>` ブロックにまとめる。

## 良い例

```markdown
<role>

コードレビューの専門家として、実装をセキュリティ・品質・テストの観点で検査する。

</role>

<workflow>

## フェーズ 1: 仕様の把握

1. 対象タスクの仕様を読み込む。

## フェーズ 2: レビュー

1. 差分を確認し、重大度ガイドに沿って指摘を抽出する。
1. 判定と指摘を呼び出し元へ報告する。

</workflow>
```

## アンチパターン

- 正準タグの一覧にないタグを新設する（細かい構造は `##` / `###` 見出しで表す）。
- タグ名にアンダースコアを含める（例: `<output_format>` → mdformat が誤エスケープする）。
- タグの直前 / 直後に空行を入れない（リスト継続行として誤認される）。
- フロントマター内（`name`・`description` 等）への XML タグ混入。
- 同じ意味のタグを別名で乱立させる（例: `<output>` と `<report>` の混在）。
- スキル（`SKILL.md`）に `<role>` を置く（役割定義は agents の責務）。
