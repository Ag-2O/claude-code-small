# Markdown 記述ガイドライン

本ガイドラインはすべての Markdown ファイルに適用する汎用スタイルを定める。
`.claude/` 配下のプロンプトファイル（エージェント定義・スキル・コマンド・`CLAUDE.md`）に固有の
XML タグ構造化ルールは、`.claude/rules/xml-tags.md` を参照する。

## 1. 言語

- Markdown ドキュメントはデフォルトで**日本語**で記述する。
- ユーザーから明示的に指示された場合のみ、他の言語を使用する。

## 2. フォーマット

### 見出し

- ATX スタイルの見出し（`#`、`##`、`###`）を使用する。Setext スタイル（`===`、`---`）は使わない。
- 見出しレベルを飛ばさない（例: `##` から `####` へ直接進まない）。
- 各見出しの前後に空行を入れる。
- 見出しの末尾に句読点を付けない。

### リスト

- 箇条書きには `-` を使う（`*` や `+` は使わない）。
- ネストしたリストは 2 スペースでインデントする。
- 段落に続くリストの前後には空行を入れる。

### コードブロック

- フェンスコードブロック（```` ``` ````）を使う。インデント式コードブロックは使わない。
- フェンスコードブロックには必ず言語識別子を付ける（例: ```` ```python ````）。

### インライン要素

- インラインコード、ファイルパス、コマンド、技術用語にはバッククォートを使う。
- 強調には `**太字**` を使う。`__太字__` は使わない。
- タイトルや軽い強調には `_イタリック_` を使う。`*イタリック*` は使わない。

### リンクと画像

- 同じ URL を繰り返す場合は参照スタイルリンクを使う。
- 画像には必ず代替テキストを付ける。

## 3. 行長と空白

- 可能な限り 1 行は 120 文字以内に収める。
- 行末に余分な空白を付けない。
- ファイルの末尾は必ず 1 つの改行で終わる。
- 段落間は空行 1 つで区切る。連続する複数の空行は使わない。

## 4. テーブル

- 必ずヘッダー行とセパレーター行を含める。
- セパレーターのダッシュは統一した形式にする（例: `---` または `:---`）。

## 5. テンプレート

- 作成するドキュメントの種類に対応するテンプレートファイルがある場合は、**必ずそれに従う**。
- テンプレートで定義されたセクション見出し、順序、構造要素をすべて保持する。
- ユーザーから明示的な指示がない限り、セクションの追加・削除・並べ替えはしない。
- プレースホルダー（例: `{{title}}`、`<!-- TODO -->`）には実際の内容を入れる。
  最終出力にプレースホルダーを残さない。

## 6. YAML フロントマター

ファイルが YAML フロントマターブロック（`---`）で始まる場合:

- 複数行の文字列には `>-` ブロックスカラーを使う。
- 同じフィールド内の文の区切りには改行を使う（空行は入れない）。

```yaml
# Good
description: >-
  最初の文。
  2 番目の文が続く。
```

## 7. アンチパターン

- 他の構文と競合する Setext スタイルの見出し（`===`、`---`）の使用。
- 同じリスト内でリストマーカーを混在させる（例: `-` と `*` の混用）。
- フェンスコードブロックへの言語識別子の省略。
- 見た目のサイズ調整を目的とした見出しレベルの飛ばし。
- 行末への余分な空白の残留。
- YAML フロントマターのブロックスカラーでの `|-` の使用（`>-` を使う）。

## 8. フォーマットと lint の対応（必須）

- **フォーマットはフックが自動実行する**: Markdown ファイルを Write / Edit すると、PostToolUse フックが
  `mdformat` を自動適用する。エージェントが手で整形する必要はない。
- **lint はゲートで実行する**: `pymarkdown` はフックでは実行しない（編集ごとの lint は診断がコンテキストを
  圧迫するため）。文書生成やワークフローの検証ゲートでまとめて実行する。具体的なコマンドと手順は
  `base-markdown` スキルの「format と lint の実行」を参照する。
- **完了前に lint 違反を 0 件にする**: 文書の作成・更新の完了報告前に lint ゲートを実行し、違反を解消する。
  問題 0 件を明示するか、残件がある場合はその理由を報告に含める。
- ユーザーが「修正不要」を明示した場合のみ、理由を添えて未修正のまま報告してよい。
- 同一問題で 3 回修正して解消できない場合は停止し、原因と次の選択肢をユーザーに確認する。
