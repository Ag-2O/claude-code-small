# 指示

クイックレビュー: $ARGUMENTS

ワークフロー（`/flow:*`）の外で、直近の小規模な変更をレビューする。
フィーチャー名やタスク ID・state.db は受け取らず、`.artifacts/` のドキュメントも参照・作成しない。
`$ARGUMENTS` はレビュー対象や重点観点の補足として扱い、無ければ作業ツリーの変更を対象とする。

コードの検査と判定・指摘の報告は `quick-reviewer` 定義の責務に従って実行される（ここでは再指示しない）。

<workflow>

1. レビュー対象を確定する。`$ARGUMENTS` で範囲・観点の指定があればそれを優先し、無ければ
   作業ツリーの変更（`git diff` / `git status` で把握）または会話で直近に実装した内容を対象とする。
1. `quick-reviewer` サブエージェントを **1 回** 起動する。対象範囲（変更ファイルのヒント・差分の所在・
   会話で合意した受け入れ観点）と重点観点を prompt に渡す。コード本体は `quick-reviewer` が
   `git diff` / `Read` で自身で読むため、コード内容自体は prompt へ複製しない。
1. `quick-reviewer` の報告（判定・指摘）をユーザーへ伝える。Warning / Block の指摘は
   `/quick:code` で修正するか、メインが直接修正する形で対応を促す。

</workflow>

<principles>

- **直近の変更を対象とする**: 作業ツリーの差分や会話で直近に実装した範囲をレビュー対象の真とする。
  state.db やフィーチャー単位の一括レビューは扱わない。
- **ドキュメントを参照しない・作成しない**: タスク仕様・サマリー・設計仕様等を前提にせず、
  review-fix タスクやレビュー結果ドキュメントも生成しない。指摘は報告で直接返す。
- **修正はその場でループしない**: 指摘はユーザーへ報告し、修正は `/quick:code` かメインの直接修正で
  別途対応する。自動で修正ループは回さない。
- **小規模に保つ**: フィーチャー単位の網羅的レビューは `/flow:review` へ誘導し、本コマンドは
  細かい変更のレビューに留める。

</principles>
