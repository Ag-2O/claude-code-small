# 指示

クイック実装: $ARGUMENTS

ワークフロー（`/flow:*`）の外で、これまでの会話で固まった内容を仕様として小規模な実装・修正を行う。
フィーチャー名やタスク ID は受け取らず、`.artifacts/` のドキュメントも参照・作成しない。
`$ARGUMENTS` は任意の補足指示として扱い、無ければ会話の文脈のみを仕様とする。

TDD 実装と検証・報告は `quick-coder` 定義の責務に従って実行される（ここでは再指示しない）。

<workflow>

1. これまでの会話から、実装・修正すべき内容を仕様として整理する。
   合意済みの要求・対象範囲・制約・メインが下した決定事項を一つにまとめる。
1. 仕様が小規模実装に足るほど明確かを確認する。曖昧で実装方針を決められない場合は、
   `quick-coder` を起動せずユーザーへ確認すること。
1. `quick-coder` サブエージェントを **1 回** 起動する。会話履歴はサブエージェントから見えないため、
   手順 1 で整理した仕様を prompt 本文に過不足なく含めること（ここでは参照すべきタスクファイルが
   存在しないため、`/flow:code` の「prompt 最小化」原則は適用しない）。対象ファイルのヒントや
   制約があれば併せて渡す。
1. `quick-coder` の報告を確認する。要決定事項・実装失敗・途中終了があれば、その場で停止して
   ユーザーへ報告し、指示を仰ぐ。要決定事項は判断後、決定を補足して `quick-coder` を再度起動する。

</workflow>

<principles>

- **会話を仕様とする**: 前工程のドキュメントは前提にせず、これまでの会話で固まった内容を仕様の真とする。
- **ドキュメントを作成しない**: requirements.md・specification.md・plans.md・summaries/phase_<phase>.md・state.db・
  タスクファイル等の成果物ドキュメントは生成・更新しない。成果はコードとテストそのものとする。
- **仕様はメインが prompt へ渡す**: サブエージェントは会話を読めないため、参照ドキュメントが無い分、
  実装に必要な仕様はメインが整理して prompt 本文へ明示的に渡す。
- **曖昧なら確認する**: 仕様が小規模実装に足るほど明確でない場合は起動前にユーザーへ確認する。
- **小規模に保つ**: 大規模・多フェーズにわたる開発は `/flow:*` ワークフローへ誘導し、本コマンドは
  細かい実装・修正の範囲に留める。

</principles>
